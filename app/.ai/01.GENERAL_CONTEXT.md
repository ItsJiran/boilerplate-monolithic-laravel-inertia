Enterprise Hybrid Monolith: Architecture & Reference Guide

This document is the Single Source of Truth for coding standards in this project.

> [!CAUTION]
> **CONTAINERIZED ENVIRONMENT ONLY**
> Never run `php artisan migrate:fresh`, `php artisan db:seed`, or other destructive database commands directly on the host machine.
> We use a Docker container environment. All database operations must be run inside the container or be handled by the container startup mechanism.

The focus of this architecture is "Write Once, Serve Both". One Controller and one Service must be able to serve the Frontend (Inertia.js) and External API (Mobile App/Integration) simultaneously without logic duplication.
1. Core Philosophy (Iron Laws)

    Agnostic Controller:

        Controllers must not care whether the client is a Browser (Inertia) or Mobile App (JSON).

        Controllers delegate output decisions to the AppResponse helper.

    Must be Stateless Service:

        Services only return data (Model/Collection/Boolean). Services MUST NOT return Inertia::render or response()->json().

    Strict DTO:

        Input to Service must use DTO.

    Smart Response Handling:

        Use the AppResponse helper to handle output. This helper will detect the request header and provide the appropriate response (JSON vs Inertia).

3. Reference Implementation (Study Case: Create Project)

Case study: User creates a new project. If via Web, redirect to index with flash message. If via API, return project JSON data.
A. The Data Transfer Object (DTO)

Same as before, no changes.
PHP

namespace App\DTO\Project;

use Illuminate\Http\Request;
use Carbon\Carbon;

readonly class CreateProjectDto
{
    public function __construct(
        public int $userId,
        public string $name,
        public ?string $description,
        public Carbon $startDate
    ) {}

    public static function fromRequest(Request $request): self
    {
        $validated = (object) $request->validated();

        return new self(
            userId: $request->user()->id,
            name: $validated->name,
            description: $validated->description ?? null,
            startDate: Carbon::parse($validated->start_date)
        );
    }
}

B. The Service Layer

Same as before. Service remains pure data.
PHP

namespace App\Services\Project;

use App\Models\Project;
use App\DTO\Project\CreateProjectDto;
use Illuminate\Support\Facades\DB;

class ProjectService
{
    public function createProject(CreateProjectDto $dto): Project
    {
        return DB::transaction(function () use ($dto) {
            return Project::create([
                'name' => $dto->name,
                'description' => $dto->description,
                'start_date' => $dto->startDate,
                'created_by' => $dto->userId,
                'status' => 'active'
            ]);
        });
    }
}

C. The Hybrid Response Helper (The Magic)

Lokasi: app/Services/Shared/AppResponse.php

Helper ini mendeteksi wantsJson().
PHP

namespace App\Services\Shared;

use Illuminate\Http\JsonResponse;
use Illuminate\Http\RedirectResponse;
use Inertia\Inertia;
use Inertia\Response as InertiaResponse;

class AppResponse
{
    /**
     * Handle sukses respon.
     * - API: Return JSON data.
     * - Web: Return Inertia Render ATAU Redirect dengan Flash Message.
     */
    public static function success(
        mixed $data = null, 
        string $message = 'Success', 
        ?string $view = null,      // Jika ingin render halaman (Inertia)
        ?string $route = null      // Jika ingin redirect (Inertia)
    ): JsonResponse|InertiaResponse|RedirectResponse
    {
        // 1. Jika Request adalah API (Mobile/Postman)
        if (request()->wantsJson()) {
            return response()->json([
                'success' => true,
                'message' => $message,
                'data'    => $data,
            ], 200);
        }

        // 2. Jika Request adalah Web (Inertia)
        
        // Kasus A: Redirect (Biasanya setelah Create/Update/Delete)
        if ($route) {
            return to_route($route)->with('success', $message);
        }

        // Kasus B: Render Halaman (Biasanya Get/Index/Show)
        if ($view) {
            // Kita bisa attach extra data (flash message) jika perlu
            return Inertia::render($view, $data);
        }

        // Default fallback (jarang terjadi)
        return back()->with('success', $message);
    }

    /**
     * Handle error respon.
     */
    public static function error(
        string $message = 'Error', 
        mixed $details = null, 
        int $status = 500
    ): JsonResponse|RedirectResponse
    {
        // 1. API: Return JSON Error
        if (request()->wantsJson()) {
            return response()->json([
                'success' => false,
                'message' => $message,
                'error_code' => $details,
            ], $status);
        }

        // 2. Web: Redirect Back dengan Flash Error
        return back()->withErrors(['message' => $message]);
    }
}

D. The Hybrid Controller

Lokasi: app/Http/Controllers/ProjectController.php

Perhatikan betapa bersihnya controller ini. Dia melayani dua "Tuan" (API & Web) sekaligus.
PHP

namespace App\Http\Controllers;

use App\Http\Controllers\Controller;
use App\Http\Requests\StoreProjectRequest;
use App\Services\Project\ProjectService;
use App\DTO\Project\CreateProjectDto;
use App\Services\Shared\AppResponse;

class ProjectController extends Controller
{
    public function __construct(
        protected ProjectService $projectService
    ) {}

    // 1. Contoh Read Data (Index)
    public function index()
    {
        // Misal service punya method getAll()
        $projects = $this->projectService->getAllProjects();

        // Hybrid Return:
        // - API: JSON list project
        // - Web: Render halaman 'Projects/Index' dengan props 'projects'
        return AppResponse::success(
            data: ['projects' => $projects],
            view: 'Projects/Index'
        );
    }

    // 2. Contoh Write Data (Store)
    public function store(StoreProjectRequest $request)
    {
        try {
            $dto = CreateProjectDto::fromRequest($request);
            $project = $this->projectService->createProject($dto);

            // Hybrid Return:
            // - API: JSON project baru + message
            // - Web: Redirect ke route 'projects.index' + Flash message
            return AppResponse::success(
                data: $project,
                message: 'Project created successfully',
                route: 'projects.index' 
            );

        } catch (\Exception $e) {
            return AppResponse::error(
                message: 'Failed to create project',
                details: $e->getMessage()
            );
        }
    }
}

4. Controller Rules (Hukum Besi Tambahan)

### 4.1 Controller Hanya Boleh:
1. **Autentikasi & authorisasi** — ambil `Auth::user()`, panggil `accessControlService`, `abort(403)`.
2. **Validasi input awal** — via FormRequest atau `$request->validate(...)`.
3. **Membaca request** — ambil input, resolve DTO dari Request.
4. **Memanggil Service** — satu atau lebih service call yang mengembalikan data.
5. **Mengembalikan response** — selalu via `AppResponse::success(...)` atau `AppResponse::error(...)`.

### 4.2 Controller DILARANG KERAS:
- ❌ Definisi query Eloquent (`Model::where(...)`, `DB::table(...)`) langsung di controller.
- ❌ Method helper pribadi (`protected function formatXxx`, `protected function buildXxx`) yang berisi business/data logic — **taruh di Service**.
- ❌ Manipulasi/transformasi data (looping, mapping, filtering) — **taruh di Service**.
- ❌ Logika kondisional bisnis (cek status, cek permission domain) — **taruh di Service**.
- ❌ Melakukan `Inertia::render(...)` atau `response()->json(...)` langsung — **gunakan `AppResponse`**.
- ❌ Duplikasi method yang sudah ada di Service — **selalu cari dulu di Service sebelum membuat method baru**.

### 4.3 Aturan Method Helper di Controller:
Jika kamu merasa perlu membuat `protected function` di dalam controller, **tanya dulu**:
- Apakah logic ini bisa masuk ke Service yang sudah ada?
- Apakah ini murni formatting response (tidak ada query/logika bisnis)?

Jika jawabannya bukan keduanya, **pindahkan ke Service**. Boleh ada `protected function` di controller **hanya** jika itu wrapper satu-baris untuk memanggil service (dan bahkan itu pun sebaiknya dihilangkan).

### 4.4 Contoh Pola yang Benar:

```php
// ✅ BENAR — Controller bersih
public function index(Request $request)
{
    $user = Auth::user();
    $allowedTenantIds = $this->accessControlService->allowedTenantIds($user);

    $data = $this->someService->getPaginatedData($allowedTenantIds, ...);

    return AppResponse::success(
        data: ['items' => $data],
        view: 'Module/Index'
    );
}

// ❌ SALAH — Ada local mapping, ada query Model langsung
public function index(Request $request)
{
    $items = Model::where('tenant_id', $tenantId)->get()->map(fn ($i) => [...]);
    return Inertia::render('Module/Index', ['items' => $items]);
}
```

### 4.5 Gunakan Method Service yang Sudah Ada:
Sebelum membuat method baru di service, **wajib cek** method yang sudah ada:
- `TenantService::getTenantOptions()` — opsi tenant untuk dropdown.
- `CustomerService::getCustomerOptions()` — opsi customer untuk dropdown.
- `CustomerValidationService::isIdentityUnique()` — cek duplikasi identitas.
- `CustomerValidationService::normalizePhoneNumber()` — normalisasi nomor HP.
- `LocationFuzzyMatcher::match()` — resolve nama lokasi ke ID.
- `AppResponse::success() / error()` — semua response keluar dari controller.

5. Service Rules (Hukum Besi Tambahan)

### 5.1 Setiap Service Punya Domain yang Jelas

Service dikelompokkan berdasarkan **domain tanggung jawab**. Method di dalam service **harus relevan** dengan domain tersebut:

| Service | Domain / Tanggung Jawab |
|---|---|
| `CustomerService` | CRUD customer, query, format detail customer |
| `CustomerIdentityService` | Buat/ubah/hapus identity (email, phone, WA) |
| `CustomerValidationService` | Rule validasi domain customer (uniqueness, normalisasi) |
| `CustomerRelationService` | Relasi antar customer (individual ↔ organization) |
| `CustomerLockService` | Status lock customer |
| `ActivityStageService` | Stage lifecycle per activity |
| `OrchestratorActivityService` | Orkestrasi alur pembuatan/update marketing activity |
| `LocationFuzzyMatcher` | Resolve nama lokasi (provinsi/kabupaten/dll) ke ID |
| `TenantService` | CRUD tenant, opsi tenant untuk dropdown |
| `NotificationService` | Kirim & kelola notifikasi |
| `AppResponse` | Format response keluar dari controller |

### 5.2 Aturan Saat Menulis Method di Service

Sebelum menulis method baru di dalam sebuah service, **wajib tanyakan kepada diri sendiri (atau kepada user)**:

1. **Apakah method ini masuk domain service ini?**
   - Contoh: Method normalisasi nomor HP **bukan** domain `CustomerService`, melainkan `CustomerValidationService`.
   - Contoh: Method fetch tenant options **bukan** domain `CustomerService`, melainkan `TenantService`.

2. **Apakah method serupa sudah ada di service lain?**
   - Cek seluruh service sebelum membuat method baru.
   - Duplikasi logic antar service adalah **bug arsitektur**.

3. **Jika method sudah di luar domain service ini** → **STOP**, dan tanyakan kepada user:
   > *"Method `[nama method]` sepertinya lebih cocok di `[NamaService]` karena [alasan domain]. Apakah saya pindahkan ke sana, atau apakah sudah ada method yang melakukan hal serupa?"*

### 5.3 Tanda-tanda Method Salah Tempat

- Method di `CustomerService` melakukan query ke tabel `tenants` → pindah ke `TenantService`.
- Method di `OrchestratorActivityService` normalisasi string → pindah ke service validasi/shared.
- Method di `CustomerIdentityService` membuat marketing activity → pindah ke `OrchestratorActivityService`.
- Method di controller melakukan business logic → pindah ke service yang sesuai.
- Method yang sama (atau nyaris sama) muncul di dua service berbeda → refactor, pilih satu owner.

### 5.4 Wajib: Paragraph Comment di Atas Class Service

Setiap file service **wajib** memiliki PHPDoc block di atas `class` yang menjelaskan:
1. **Apa** yang dilakukan service ini (satu kalimat ringkas).
2. **Domain** mana yang menjadi tanggung jawabnya.
3. **Dependency utama** yang diandalkan (opsional, jika tidak obvious).

```php
// ✅ BENAR — ada paragraph comment yang jelas
/**
 * Mengelola pembuatan, pembaruan, dan penghapusan data identitas pelanggan
 * (email, nomor telepon, WhatsApp).
 *
 * Domain: Customer Identity
 * Bergantung pada CustomerValidationService untuk normalisasi & validasi uniqueness.
 */
class CustomerIdentityService { ... }

// ✅ BENAR — service orkestrasi
/**
 * Mengorkestrasikan alur pembuatan dan transisi status Marketing Activity.
 * Menggabungkan ActivityStageService, CustomerLockService, dan broadcast event
 * dalam satu titik masuk yang konsisten.
 *
 * Domain: Marketing Activity Lifecycle
 */
class OrchestratorActivityService { ... }

// ❌ SALAH — tidak ada keterangan sama sekali
class CustomerService { ... }
```

Jika kamu diminta menambahkan method ke sebuah service yang **belum** punya paragraph comment, **tambahkan juga comment-nya** sekalian.

Jika sebuah service **butuh** fungsionalitas dari service lain, gunakan **injeksi dependency** — bukan duplikasi:

```php
    {
        $normalized = $this->validationService->normalizePhoneNumber($data['identity_value']);

    }

// ❌ SALAH — salin-tempel logic normalisasi ke service lain
class SomeOtherService
{
    protected function normalizePhone(string $phone): string
    {
        // logic yang sama persis dengan CustomerValidationService
    }
}
```

### 5.6 Aturan Orchestrator Service (Hukum Besi)

**Orchestrator service (`Orchestrator*`) adalah *konduktor*, bukan implementor.**

Sebuah class Orchestrator **hanya boleh**:
- Memanggil method dari service lain (via injeksi dependency).
- Mengatur urutan/alur panggilan tersebut dalam satu `DB::transaction()`.
- Melakukan branching sederhana (`if/match`) untuk memilih alur yang tepat.

Sebuah class Orchestrator **dilarang keras**:
- Memiliki **private/protected helper methods** yang berisi business logic.
- Melakukan query database langsung (`DB::table(...)`, `Model::create(...)`) **kecuali** untuk entitas utama yang menjadi "output" dari orkestrasi tersebut.
- Menduplikasi logic yang sudah ada di service terdedikasi.

```php
// ✅ BENAR — orchestrator hanya memanggil service lain
class OrchestratorActivityService
{
    public function createActivityForIndividual(array $data): array
    {
        return DB::transaction(function () use ($data) {
            $this->activityValidationService->validateIndividualActivityRequest($data);
            $this->activityValidationService->validateMarketingUserForTenant($data);

            $activity = MarketingActivity::create([...]); // entitas output utama = OK

            $stage = $this->activityStageService->createInitialStage($activity); // delegasi
            $lock  = $this->customerLockService->lockCustomer(...);              // delegasi

            $this->broadcastService->broadcastActivityCreated($activity, $stage);

            return compact('activity', 'stage', 'lock');
        });
    }
}

// ❌ SALAH — orchestrator punya helper method sendiri
class OrchestratorActivityService
{
    private function buildInitialStage(MarketingActivity $activity): MarketingActivityStage
    {
        // Logic ini harusnya di ActivityStageService!
        return MarketingActivityStage::create([...]);
    }
}
```

> Jika kamu terdorong untuk menulis private method di dalam Orchestrator, itu sinyal kuat bahwa logika tersebut harus berada di **service domain yang sudah ada** atau memerlukan **service baru**.

6. Workflow Decision Tree

Saat membuat Controller method, gunakan panduan ini untuk parameter AppResponse::success:

    Apakah ini halaman GET (Menampilkan data)?

        Gunakan param view: 'Nama/Komponen/Inertia'.

        Isi data dengan array props yang dibutuhkan Vue/React.

        Hasil: API dapat JSON, Web dapat Halaman.

    Apakah ini aksi POST/PUT/DELETE (Modifikasi data)?

        Gunakan param route: 'nama.route.tujuan'.

        Isi message untuk notifikasi sukses.

        Hasil: API dapat JSON konfirmasi, Web melakukan Redirect + Flash Message.

    Apakah terjadi Error?

        Gunakan AppResponse::error.

        Hasil: API dapat status 500/400, Web akan redirect back dengan error bag (bisa ditampilkan di UI).

7. Frontend Guidelines (Inertia.js)

    Structure & State Management:
        - Gunakan `useForm` untuk semua state yang akan dikirim ke backend.
        - Hindari `useState` terpisah untuk input form (seperti dynamic inputs). Masukkan langsung ke dalam `useForm` data (array/object).
        - Update state menggunakan `setData`.

    Form Submission:
        - JANGAN PERNAH menyertakan properti `data` manual di argumen kedua `post/put/patch`. 
        - Inertia secara otomatis mengirimkan object `data` dari `useForm`.
        - Salah: `post(url, { data: { ... } })`
        - Benar: `post(url)` (Pastikan `data` sudah terupdate via `setData` sebelumnya logic state sudah ada di `data`).

8. Enum Usage & Storage Pattern

    Database Storage:
        - Enum SELALU disimpan sebagai STRING di database (kolom VARCHAR/TEXT).
        - JANGAN gunakan casting enum di Model (`protected $casts`).
        - Alasan: Jika enum berubah di masa depan (nilai atau case dihapus/diubah), data lama tidak akan error/corrupt.

    Controller & Service:
        - Gunakan Enum untuk type safety dan validasi di Controller dan Service layer.
        - Contoh: `IdentityType::EMAIL->value`, `CustomerType::B2B->value`.

    Model Methods:
        - Method di Model tetap menerima dan mengembalikan STRING.
        - Jika perlu konversi ke Enum, lakukan secara eksplisit di Controller/Service.
        
    Contoh Pattern:
        ```php
        // ❌ SALAH - Jangan cast di Model
        protected $casts = [
            'customer_type' => CustomerType::class,
        ];
        
        // ✅ BENAR - Store as string
        $customer->customer_type = CustomerType::B2B->value; // 'B2B'
        
        // ✅ BENAR - Convert explicitly when needed
        $enumValue = CustomerType::from($customer->customer_type);
        ```
